## 一、插件目录结构 + 一个可迁移的代码阅读直觉

`extensions/anthropic/` 下有 28 个文件。但通过文件名能快速识别"哪两个最关键"：

| 文件 | 通用程度 | 解读 |
|---|---|---|
| `package.json`、`tsconfig.json`、`index.ts` | 全 npm/TS 项目通用 | 不是 OpenClaw 特有 → 不是定义"什么是插件"的关键 |
| **`openclaw.plugin.json`** | 带项目名前缀 | 项目自己规定的"契约清单" → 几乎一定是关键 ⭐ |
| **`register.runtime.ts`** | 动词性文件名 | 文件名本身在叫："我是用来注册到运行时的" ⭐ |

📌 **可迁移的代码阅读直觉**：
- **带项目名前缀的文件**（`openclaw.*`）→ 项目自己定义的"契约"
- **动词性文件名**（`register.*`、`bootstrap.*`、`setup.*`）→ 项目设计的"接缝点"

## 二、第一个特殊文件：`openclaw.plugin.json`（声明 / 元数据）

### 字段按"语义角色"分成 5 组（看陌生 JSON 配置的通用技巧）

```
1. "我是谁"             id, activation, enabledByDefault
2. "我能做什么"  ⭐      providers, providerDiscoveryEntry,
                      modelSupport.modelPrefixes, contracts
3. "我怎么和外部世界打交道" providerEndpoints, providerRequest, cliBackends
4. "用户怎么登录我"      providerAuthEnvVars, providerAuthChoices
5. "杂项细节"           modelIdNormalization, modelPricing, configSchema
```

本节课只深入了第 1、2 组。第 3、4、5 组进盲区清单，后续讲到具体场景时再回来看。

### 第 1 组：身份与开关

```json
"id": "anthropic"             ← 全局唯一身份证
"activation.onStartup": false ← 启动时不要立即激活
"enabledByDefault": true      ← 但默认是开启状态
```

⚠️ `onStartup: false` 但 `enabledByDefault: true` 看似矛盾，实则不矛盾：
- `enabledByDefault` = "我是否计入可用清单"（开/关）
- `onStartup` = "我什么时候被加载"（积极/懒）
- 翻译："我可以用，但你别一启动就拉我，等真有人用 claude-xxx 模型时再加载" → **懒加载（lazy loading）**

### 第 2 组：能力声明（这一组是这个文件的"魂"）

```json
"providers": ["anthropic"],
"providerDiscoveryEntry": "./provider-discovery.ts",
"modelSupport": { "modelPrefixes": ["claude-"] },
"contracts": { "mediaUnderstandingProviders": ["anthropic"] }
```

- `providers` —— 我提供 "anthropic" 这种 provider
- `modelPrefixes` —— **路由匹配规则**：用户用 `claude-*` 时交给我
- `providerDiscoveryEntry` —— 指向代码入口的"路标"
- `contracts` —— 我还实现了"媒体理解"契约

## 三、关键路由代码：网关怎么用 modelPrefixes

`src/plugins/bundled-runtime-deps-selection.ts:373`：

```typescript
return (manifest.modelSupport?.modelPrefixes ?? []).some(
  (prefix) => modelId.startsWith(prefix)
);
```

**关键观察**：输入是 `modelId` 而不是 `userMessage`——主程序拿到的已经是"干净的模型 ID 字符串"，不是用户的原始消息。

继续追溯（同文件 376-396 行），`modelId` 是从 `OpenClawConfig` 里 **parse** 出来的，**不是从消息流里"提取"的**。

### 控制信道 vs 内容信道（重要架构概念）

```
┌──────────────────────────────────────────────────────────────┐
│  用户在 Discord 里                                            │
│                                                              │
│  控制信道（结构化）：           内容信道（自由文本）：         │
│  • slash 命令 /model ...       "帮我写首诗"                  │
│  • CLI --model=...             "用唐诗的格式"                │
│  • UI 下拉框                                                 │
│  • 配置文件 default            （任意自然语言）              │
│           │                              │                   │
│           ↓                              ↓                   │
│      主程序读 ← 路由用            主程序不读 ← 透传给 AI 模型 │
└──────────────────────────────────────────────────────────────┘
```

主程序**对消息体内容是"盲"的**——它就像快递公司，读包裹上的地址标签（控制信道），但绝不打开包裹内容（内容信道）。

📌 **核心口诀**：
- **"网关只路由，插件只翻译，思考只在模型里。"**
- **"网关只看标签，不开包裹。"**

## 四、第二个特殊文件：`register.runtime.ts`（代码 / 实现）

### 灵魂只在最后 5 行（第 594-598 行）

```typescript
export function registerAnthropicPlugin(api: OpenClawPluginApi): void {
  api.registerCliBackend(buildAnthropicCliBackend());
  api.registerProvider(buildAnthropicProvider());
  api.registerMediaUnderstandingProvider(anthropicMediaUnderstandingProvider);
}
```

整个文件 598 行，前 590 行都在准备"要挂上去的东西"，**最后这 5 行才是真正"挂"的动作**。

### 核心机制：控制反转（Inversion of Control, IoC）

不是主程序去读插件、解析、决定怎么用——而是**主程序借出 `api` 对象，插件主动调 `api.register*` 把自己挂上去**。

🍔 **类比**：餐厅老板（主程序）把菜单本（`api`）递给新厨师（插件），让厨师自己写"我能做啥"。
- 老板**不会**亲自翻厨师的简历挑能做的菜
- 厨师写下：`api.addDish('巨无霸')`、`api.addDish('薯条')`

→ 这就叫"**控制反转**"：注册的主动权在插件。

### 挂上去的"能力对象"骨架（buildAnthropicProvider 返回值）

```typescript
return {
  id: "anthropic",                              // 我是谁
  envVars: ["ANTHROPIC_OAUTH_TOKEN", "..."],
  auth: [...],                                  // 登录方式（一次性动作）
  resolveDynamicModel: (ctx) => {...},          // 解析模型 ID
  resolveSyntheticAuth: (...) => {...},         // 取登录态
  wrapStreamFn: wrapAnthropicProviderStream,    // 包装流式响应（每次对话都用）
  fetchUsageSnapshot: async (...) => {...},     // 查 quota
  // ...十几个方法
};
```

### 设计模式：Provider 接口 / Strategy 模式

插件不是"继承一个基类"，而是"返回一个**符合特定形状的对象**"。
- 主程序只关心这个对象上有没有 `resolveDynamicModel`、`wrapStreamFn` 等方法
- 不同 provider（OpenAI、Google、Anthropic）返回的对象**结构一样、方法不同**
- 这是"用统一接口适配多种后端"的精髓

## 五、把三个角色串成完整故事

```
┌──────────────────────────────────────────────────────────────┐
│ 主程序启动时...                                                │
│                                                              │
│  1️⃣ 扫描 extensions/ 目录                                    │
│      → 找到 extensions/anthropic/openclaw.plugin.json        │
│      → 读出"声明"：管 claude- 前缀，入口在 register.runtime.ts │
│                                                              │
│  2️⃣ 主程序需要用 anthropic 时（懒加载）                       │
│      → 加载 register.runtime.ts                              │
│      → 调用其 export 出来的 registerAnthropicPlugin(api)      │
│                                                              │
│  3️⃣ 插件主动登记                                              │
│      → api.registerProvider({ id, auth, resolveDynamicModel,│
│                              wrapStreamFn, ... })            │
│      → 主程序拿到一个"能力对象"                                │
│                                                              │
│  4️⃣ 路由请求到来时                                            │
│      → 用户请求 modelId="claude-opus-4-6"                    │
│      → 按 modelPrefixes 匹配 → 命中 anthropic                │
│      → 调用 anthropicProvider.resolveDynamicModel(ctx)       │
│      → 调用 anthropicProvider.wrapStreamFn(stream)            │
│      → 真正的请求飞向 api.anthropic.com                      │
└──────────────────────────────────────────────────────────────┘
```

## 六、设计哲学：声明 vs 实现 / 让"机器读"和"机器跑"分开

### 关键判断：plugin.json 的 authChoices 与 register.runtime.ts 的 auth 数组重复吗？

**不重复，各有所司**：

| | `plugin.json` 的 `providerAuthChoices` | `register.runtime.ts` 的 `auth: [...]` |
|---|---|---|
| 角色 | **菜单板** —— 给配置 UI / CLI / 文档生成器**读**的元数据 | **厨师手册** —— 用户点了某款后**真正执行**的代码 |
| 类比 | 餐厅门口的菜单（顾客读、外卖员读、营养师读……） | 厨房里"巨无霸怎么做"的步骤手册 |

⚠️ **重要细节**：`register.runtime.ts` 里的 auth 数组**只在登录这一次性动作**时跑，**不是**每次对话调用 AI 时跑。每次对话用的是同文件里的 `wrapStreamFn`、`resolveSyntheticAuth` 等其他字段。

### 砍掉哪个会丢什么？

- 砍 `register.runtime.ts` 里的 auth → **程序起不来**（缺执行代码）
- 砍 `plugin.json` 里的 authChoices → 仅缺"引导式 UI 素材"，代码本身能跑

### 为什么这种"看似重复"的设计是对的？

| 直觉表达 | 工程术语 |
|---|---|
| "两种目的的实现分离" | **关注点分离（Separation of Concerns）** |
| "前者引导用户配置 / 后者运行态实现" | **声明式 vs 命令式（Declarative vs Imperative）** |
| "互不干扰" | **正交设计（Orthogonal Design）** |

最深一层：**"让机器读和机器跑分开"**。

### "JSON 天然多用途"的真实含义（用户追问的关键澄清）

| | `plugin.json` | `register.runtime.ts` |
|---|---|---|
| 本质 | **数据快照**（冻结的事实） | **代码**（待执行的动作）|
| 怎么"读" | `JSON.parse(file)` —— **一行** | 要么编译运行，要么 AST 解析 |
| 谁能读 | 任何语言、任何工具——Node、Python、Web、bash | 只有能跑 Node.js / TS 编译器的环境 |

**4 个工具想拿同一个字段的成本对比**：

塞 JSON：每个工具 1 行 `JSON.parse`，4 个工具共 4 行。**任何语言都能玩**。
塞代码：每个工具都得装 TS 编译器、处理 import 链、调用函数、mock 上下文……**非 JS 语言直接出局**。

📌 **一句话**：
> "**JSON 是数据，数据没有立场——谁来读都返回同样的结果。**
> **代码是行为，行为有'执行环境'立场——要从行为反推出事实，每多一种用途都得新写提取逻辑。**"

→ 所以 OpenClaw 的策略：**事实**（id / 支持前缀 / 几种登录方式）放 `plugin.json`，**行为**（怎么登录、怎么包装请求流）放 `register.runtime.ts`。

## 用户复述与反馈

### 互动 #1：插件骨架文件识别
**用户回答**：openclaw.plugin.json + register.runtime.ts。理由：其他文件名（package.json、index.ts、tsconfig.json）通用，更像插件本身的文档；这两个文件名特别。
**反馈**：✅ 完全正确，推理路径优秀——"用文件名的非通用性反推它的项目特异性"是个非常实用的代码阅读直觉。

### 互动 #2：插件发现机制 A 还是 B？
**用户回答**：B（插件主动声明清单）。理由：分层清晰；自动扫描的问题在于不同插件编写规范可能有出入；不如提供独立可控的"插件协议清单"稳定管理。
**反馈**：✅ 答案正确，理由触及"契约先行（contract-first）"的工程原则——对应 Convention over Configuration vs Explicit Manifest。

### 互动 #3：哪个字段直接负责路由匹配？
**用户回答**：`modelSupport.modelPrefixes`
**反馈**：✅ 完全正确。

⚠️ **本节课最大认知纠错**：
用户在推理时把"主程序"想象成了一个"AI"——以为主程序需要"理解" `claude-opus-4-6` 是什么、不认识时会触发互联网搜索 skill 来"补充认知"。

实际：**主程序不"思考"任何东西，它是个机械的字符串匹配器**。AI 在插件后面的 provider 那里。

被纠正后，用户**自己推导出三条理由**解释为什么"主程序不能用 AI 思考"：
1. AI 上套 AI 无限循环
2. 把确定性的东西固化下来，让主程序稳定
3. 不确定性反馈来指导下一步运行 → 难以想象怎么设计程序

→ 直接对应 AI 工程的三个核心术语：**递归不确定性**、**确定性核心 + 概率边缘（Deterministic Core / Probabilistic Edge）**、**"Don't put a coin flip in the critical path"**。

**核心口诀**：
- "网关只路由，插件只翻译，思考只在模型里。"
- "网关只看标签，不开包裹。"
- "AI 是工具，不是大脑。"

### 用户主动追问 #2：JSON 天然多用途到底是什么意思？
