## 可迁移的代码阅读直觉

`extensions/anthropic/` 下有 28 个文件。但通过文件名可以快速识别"哪两个最关键"：

| 文件名特征 | 解读 |
|---|---|
| `package.json`、`tsconfig.json` 等通用文件 | 不是项目特异性的关键 |
| **带项目名前缀**（如 `openclaw.plugin.json`） | 项目自己规定的"契约清单" → 几乎一定是关键 |
| **动词性文件名**（`register.*`、`bootstrap.*`、`setup.*`） | 项目设计的"接缝点" → 几乎一定是关键 |

→ 这两条直觉能帮你**快速看陌生项目时挑出最关键的几个文件**。

## 第一个文件：`openclaw.plugin.json`（声明 / 元数据）

110 行 JSON 字段密集。看陌生 JSON 配置的通用技巧：**永远先按"语义角色"分组，不要逐字段读**。

```
1. "我是谁"                id, activation, enabledByDefault
2. "我能做什么"   ⭐         providers, providerDiscoveryEntry,
                         modelSupport.modelPrefixes
3. "我怎么和外部世界打交道"   providerEndpoints, providerRequest
4. "用户怎么登录我"          providerAuthEnvVars, providerAuthChoices
5. "杂项细节"              modelIdNormalization, modelPricing
```

第 2 组（能力声明）是这个文件的"魂"。`modelSupport.modelPrefixes: ["claude-"]` 是路由匹配规则——用户请求 `claude-*` 时交给这个插件。

### 一个看似矛盾、实则不矛盾的字段组合

```json
"activation.onStartup": false  ← 启动时不要立即激活
"enabledByDefault": true       ← 但默认是开启状态
```

- `enabledByDefault` = 我是否计入可用清单（开/关）
- `onStartup` = 我什么时候被加载（积极/懒）
- 翻译："我可以用，但你别一启动就拉我，等真有人用 claude-xxx 模型时再加载" → **懒加载（lazy loading）**

## 关键路由代码 + 推导出的通用架构概念

`src/plugins/bundled-runtime-deps-selection.ts:373`：

```typescript
return (manifest.modelSupport?.modelPrefixes ?? []).some(
  (prefix) => modelId.startsWith(prefix)
);
```

**关键观察**：输入参数叫 `modelId`，不是 `userMessage`——主程序拿到的已经是"干净的模型 ID"，不是用户的原始消息。继续追溯，`modelId` 来自 `OpenClawConfig`（**配置对象**），是从配置里 parse 出来的，**不是从消息流里"提取"的**。

### 推导出的通用架构概念：双信道分离

```
控制信道（结构化）           内容信道（自由文本）
• slash 命令 /model ...     "帮我写首诗"
• CLI --model=...           （任意自然语言）
• 配置文件 default
       │                              │
       ↓                              ↓
   主程序读 ← 路由用            主程序不读 ← 透传给 AI 模型
```

→ **主程序对消息体内容是"盲"的**。它就像快递公司：读包裹上的地址标签（控制信道），但绝不打开包裹内容（内容信道）。

📌 **核心口诀**：
- **"网关只路由，插件只翻译，思考只在模型里。"**
- **"网关只看标签，不开包裹。"**

### 一个深刻的衍生原理：AI 应用工程的第一性原理

这种"主程序不思考"的设计，背后是 AI 应用工程的根本约束：

| 直觉表达 | 工程术语 |
|---|---|
| "AI 上套 AI 无限循环" | **递归不确定性** |
| "把确定性的东西固化下来" | **确定性核心 + 概率边缘** |
| "不确定性反馈不能指导下一步" | "Don't put a coin flip in the critical path" |

一句话总结：**"AI 是工具，不是大脑。整个系统的大脑要是确定性的代码；AI 只是大脑指派去完成『需要思考』那一小段任务的承包商。"**

## 第二个文件：`register.runtime.ts`（代码 / 实现）

整个文件 598 行，但**真正干"挂钩"那件事的只有最后 5 行**：

```typescript
export function registerAnthropicPlugin(api: OpenClawPluginApi): void {
  api.registerCliBackend(buildAnthropicCliBackend());
  api.registerProvider(buildAnthropicProvider());
  api.registerMediaUnderstandingProvider(anthropicMediaUnderstandingProvider);
}
```

前 590 行都在准备"要挂上去的东西"，最后这 5 行才是"挂"的动作。

### 关键机制：控制反转（Inversion of Control, IoC）

不是主程序去读插件、解析、决定怎么用——而是 **主程序借出 `api` 对象，插件主动调 `api.register*` 把自己挂上去**。

🍔 **类比**：餐厅老板（主程序）把菜单本（`api`）递给新厨师（插件），让厨师自己写"我能做啥"。
- 老板**不会**亲自翻厨师简历挑能做的菜
- 厨师写下：`api.addDish('巨无霸')`、`api.addDish('薯条')`

### 挂上去的"能力对象"长这样

```typescript
return {
  id: "anthropic",
  envVars: ["ANTHROPIC_OAUTH_TOKEN", "..."],
  auth: [...],                              // 登录方式（一次性动作）
  resolveDynamicModel: (ctx) => {...},      // 解析模型 ID
  wrapStreamFn: wrapAnthropicProviderStream, // 包装流式响应（每次对话都用）
  fetchUsageSnapshot: async (...) => {...}, // 查 quota
  // ...十几个方法
};
```

→ 这是 **Provider 接口 / Strategy 模式**：插件不"继承基类"，而是"返回一个**符合特定形状的对象**"。不同 provider（OpenAI、Google、Anthropic）返回的对象**结构相同、方法不同**。

```
│                                                              │

│                                                              │

└──────────────────────────────────────────────────────────────┘
```

## 设计哲学：声明 vs 实现 / 让"机器读"和"机器跑"分开

### 关键判断：plugin.json 里的 `providerAuthChoices` 与 register.runtime.ts 里的 `auth` 数组——重复吗？

**不重复，各有所司**：

| | `plugin.json` 的 `providerAuthChoices` | `register.runtime.ts` 的 `auth: [...]` |
|---|---|---|
| 角色 | **菜单板** —— 给配置 UI / CLI / 文档生成器**读**的元数据 | **厨师手册** —— 用户点了某款后**真正执行**的代码 |
| 类比 | 餐厅门口的菜单（顾客、外卖员、营养师都能读） | 厨房里的步骤手册（只能让厨师执行） |

⚠️ 注意：`register.runtime.ts` 里的 auth 数组**只在登录这一次性动作**时跑，**不是**每次对话调用 AI 时跑。每次对话用的是同文件里的 `wrapStreamFn`、`resolveSyntheticAuth` 等其他字段。

### 设计哲学层

| 直觉表达 | 工程术语 |
|---|---|
| "两种目的的实现分离" | **关注点分离（Separation of Concerns）** |
| "前者引导用户配置 / 后者运行态实现" | **声明式 vs 命令式（Declarative vs Imperative）** |
| "互不干扰" | **正交设计（Orthogonal Design）** |

最深一层：**"让机器读和机器跑分开"**。

### 为什么 JSON 比 TS 更适合"声明"——多用途的真实含义

| | `plugin.json` | `register.runtime.ts` |
|---|---|---|
| 本质 | **数据快照**（冻结的事实） | **代码**（待执行的动作）|
| 怎么"读" | `JSON.parse(file)` —— 一行 | 要么编译运行，要么 AST 解析 |
| 谁能读 | 任何语言、任何工具 | 只有能跑 Node.js / TS 编译器的环境 |

**4 个工具想拿同一个字段的成本对比**：

塞 JSON：配置 UI、CLI、文档生成器、安全审计 → 每个 1 行 `JSON.parse`，共 4 行。**任何语言都能玩**。
塞代码：每个工具都得装 TS 编译器、处理 import 链、调用函数、mock 上下文……**非 JS 语言直接出局**。

📌 **一句话**：

> **"JSON 是数据，数据没有立场——谁来读都返回同样的结果。**
> **代码是行为，行为有'执行环境'立场——要从行为反推出事实，每多一种用途都得新写提取逻辑。"**

## 5 条 takeaway（带走这 5 条够了）

读完这一节课你应该能回答：

1. **OpenClaw 这类项目的插件由两份文件组成**：`*.plugin.json`（声明 / 元数据）+ `register.runtime.ts`（代码 / 实现）。前者**让多种工具读**，后者**让运行时执行**。
2. **挂钩机制 = 控制反转（IoC）**：主程序借出 `api`，插件主动调 `api.register*` 登记。
3. **接口风格 = Strategy / Provider 模式**：返回一个符合特定形状的对象，不同后端"结构同、方法异"。
4. **架构原则 = 控制信道与内容信道分离**：网关只读控制信道（路由/调度），内容信道（用户消息）直接透传给 AI。
5. **设计哲学 = "让机器读和机器跑分开"**：事实塞 JSON（多用途零成本），行为塞代码（必须执行）。

这五条都是**可迁移的**——VS Code 扩展、Chrome 扩展、npm 包……几乎所有插件化系统都遵循同样的模式。
