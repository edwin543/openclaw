> **这是一份"分享版"精简笔记**——由真实学习对话浓缩而成，去掉了个人化细节，保留了方法论闪光点与可迁移的洞察。

> 学习目标：用一节课时间建立 OpenClaw 项目的"鸟瞰图"——是什么项目、技术栈、目录布局、设计哲学。

## 一句话定位

OpenClaw 是一个**跑在用户自己设备上的"个人 AI 助手"**，本质上是一个**多频道 AI 网关（multi-channel AI gateway）**。

它把"已经在用的聊天工具（24+ 个）"作为信道，把"AI 模型（多种 provider）"作为大脑，把"能做事的能力（plugins / skills / MCP）"作为手脚——三者通过 gateway 连接起来。

## 三条核心设计哲学

理解这三条，再看任何代码都有"上帝视角"。

### 1. 核心精简 + 插件化（Plugin-first）
> "Core stays lean; optional capability should usually ship as plugins."

主仓库本身只放骨干，**任何具体能力（接哪家 AI、对接哪个聊天工具、什么技能）都走插件路径**。这就是为什么 `extensions/` 下有 126 个并列子目录——每个都是独立的 npm 包。

### 2. 终端优先（Terminal-first）
首选用法是终端 `openclaw onboard` 启动。不做"傻瓜式 GUI 封装"——因为 AI 助手能"真的做事"，权限、认证、安全决策必须让用户清楚地看到。

### 3. 安全是头等公民
证据密度异常高：根目录就有 `SECURITY.md`、`INCIDENT_RESPONSE.md`、`Dockerfile.sandbox`、`zizmor.yml`（GitHub Actions 安全审计配置）……一个开源项目这么重视安全工程是少见的。

## 顶层目录功能分区图

```
openclaw/
├── src/              ← 核心代码（主战场）
├── packages/         ← 内部 npm 包（4 个：sdk、plugin-sdk 等）
├── extensions/       ← 插件库（126 个并列目录，每个是独立 npm 包）
├── apps/             ← 多端原生应用（Android Kotlin / iOS macOS Swift）
├── ui/               ← Web 前端（Vite）
├── skills/           ← 内置技能
├── docs/             ← 文档
└── security/、test/、scripts/ ……
```

📌 **关键观察**：这是个**多语言、多平台、大型 monorepo**——不是单文件玩具。但结构非常有规律，掌握"目录分区"就能快速定位。

## 一道闪光时刻：判断题 + 概念澄清

### 判断题
> 假如要"让 OpenClaw 支持一个新的 AI 模型 'XYZ-AI'"，最有可能要修改 / 新增哪个目录的代码？
>
> A. `src/agents/`　B. `src/channels/`　C. `extensions/`　D. `packages/sdk/`

**答案：C**。证据：`extensions/anthropic/`、`extensions/openai/` 都是独立完整的 npm 包，结构高度一致——接新模型确实就是"新建一个 extensions/xxx 包"。

### 顺便澄清一个普遍混淆：Plugin vs Skill

学习过程中很容易把"插件"和"skill"混用。在 OpenClaw 里这两个是**不同粒度**的概念：

| 概念 | 是什么 | 粒度 | 例子 |
|---|---|---|---|
| **Plugin / Extension** | 对接外部"基础设施"的代码模块 | 粗粒度 | 接 Anthropic 模型、接 Discord 频道、用 Azure 做 TTS |
| **Skill** | AI 能完成的某种具体任务 | 细粒度 | 创建 Word 文档、画图表 |

#### 麦当劳类比（用于巩固区分）

- **Plugin / Extension** = 餐厅里的**柜台机器**（点餐机、咖啡机）→ 基础设施
- **Skill** = 菜单上的**具体食物**（巨无霸、薯条）→ 具体能产出的东西
- **Agent** = **厨师**（编排）
- **Channels** = **门口传菜员**（与用户接触的窗口）

→ 加新 AI 模型 = 加一台点餐机 = **Plugin**（Extension）
→ 加"写诗"能力 = 加一道新菜 = **Skill**

## 可迁移的洞察口诀

这两句口诀适用于一切"接入多种 AI 后端 / 多种聊天工具"的项目设计——不只是 OpenClaw：

> 1. **"网关只路由，插件只翻译，思考只在模型里。"**
> 2. **"网关只看标签，不开包裹。"**

第一句说的是分层：**主程序是确定性代码、AI 只放在叶子节点**。
第二句说的是协议：**控制信道（模型选择/命令）走主程序读，内容信道（用户消息）直接透传给后端模型**。

这两句的更深层原理：

| 学习者直觉表达 | 工程术语 |
|---|---|
| "AI 上套 AI 无限循环" | **递归不确定性**——错误会沿着调用链放大，且无法证明终止 |
| "把确定性的东西固化下来，让主程序稳定" | **确定性核心 + 概率边缘（Deterministic Core / Probabilistic Edge）**——业界共识 |
| "不确定性反馈来指导下一步运行难以想象" | **控制流不能依赖随机性**——经典原则："Don't put a coin flip in the critical path" |

一句话总结：**"AI 是工具，不是大脑。整个系统的大脑要是确定性的代码；AI 只是大脑指派去完成『需要思考』那一小段任务的承包商。"**

## 一个实用的代码阅读直觉

看陌生项目时，识别"哪些文件最关键"：

| 文件名特征 | 解读 |
|---|---|
| `package.json`、`tsconfig.json` 等通用文件 | 不是项目特异性的关键 |
| **带项目名前缀**（如 `openclaw.plugin.json`） | 项目自己规定的"契约清单" → 几乎一定是关键 |
| **动词性文件名**（`register.*`、`bootstrap.*`、`setup.*`） | 项目设计的"接缝点" → 几乎一定是关键 |

下节课就会用这个直觉切入：从 `openclaw.plugin.json` + `register.runtime.ts` 看插件是怎么挂到主程序上的。
