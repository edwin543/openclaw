## 一、一句话定位

OpenClaw 是一个**跑在用户自己设备上的"个人 AI 助手"**，本质上是一个**多频道 AI 网关（multi-channel AI gateway）**。

- 它不是聊天框 UI，而是把"已经在用的聊天工具（24+ 个）"作为信道，把 AI 模型（多种 provider）作为大脑，把"能做事的能力"（plugins / skills / MCP）作为手脚——三者通过这个 gateway 连接起来。
- 官方一行话：*Multi-channel AI gateway with extensible messaging integrations*。
- 项目演化历史（来自 VISION.md）：Warelay → Clawdbot → Moltbot → OpenClaw。

## 二、三条核心设计哲学

理解这三条，再看任何代码都有"上帝视角"：

### 1. 核心精简 + 插件化（Plugin-first）

> "Core stays lean; optional capability should usually ship as plugins."

主仓库本身只放骨干，**任何具体能力（接哪家 AI、对接哪个聊天工具、什么技能）都走插件路径**。这就是为什么 `extensions/` 下有 126 个并列的子目录——每个目录就是一个独立的 npm 包，对应一个具体外部能力。

### 2. 终端优先（Terminal-first）

首选用法是终端 `openclaw onboard` 启动。不做"傻瓜式 GUI 封装"——因为 AI 助手能"真的做事"，权限、认证、安全决策必须让用户清楚地看到。

### 3. 安全是头等公民

证据密度异常高：根目录就有 `SECURITY.md`、`INCIDENT_RESPONSE.md`、`.detect-secrets.cfg`、`.semgrepignore`、`security/` 目录、`Dockerfile.sandbox`、`zizmor.yml`（GitHub Actions 安全审计配置）……一个开源项目这么重视安全工程是少见的。

## 三、技术栈一览

| 层 | 技术 |
|---|---|
| 核心运行时 | Node.js 22.14+（推荐 Node 24） |
| 主语言 | TypeScript |

| 版本号 | `2026.4.27`（用日期做版本号 → 频繁 release，活跃开发） |

📌 **关键观察**：这是一个**多语言、多平台、大型 monorepo**——不是单文件玩具。但它结构非常有规律，掌握"目录分区"就能快速定位。

## 五、本次的"轻动手"：判断题与验证

### 判断题

> 假如要"让 OpenClaw 支持一个新的 AI 模型 'XYZ-AI'"，最有可能要修改 / 新增哪个目录的代码？
> A. `src/agents/`  B. `src/channels/`  C. `extensions/`  D. `packages/sdk/`

### 用户答案

**C**。理由：openclaw 用插件化思路设计整个项目，对接新 AI 大模型相当于新增一种"对接 AI 的 SKILL"。

### 验证结果

✅ **答案 C 正确**。证据：

- `extensions/anthropic/` 是一个独立完整的 npm 包：有自己的 `package.json`、`tsconfig.json`、测试文件
- 关键文件 `openclaw.plugin.json`（插件清单）+ `register.runtime.ts`（注册到运行时的入口）
- `package.json` 里 name 是 `@openclaw/anthropic-provider`，description 是 `OpenClaw Anthropic provider plugin`
- `extensions/openai/` 结构高度类似 → 接新模型确实就是"新建一个 extensions/xxx 包"

### 概念澄清（用词混淆点）

用户回答里把"插件"说成了"skill"。在 OpenClaw 里这两个是**不同粒度**的概念：

| 概念 | 是什么 | 粒度 | 例子 |
|---|---|---|---|
| **Plugin / Extension** | 对接外部"基础设施"的代码模块 | 粗粒度 | 接 Anthropic 模型、接 Discord 频道、用 Azure 做 TTS |
| **Skill** | AI 能完成的某种具体任务 | 细粒度 | 创建 Word 文档、画图表 |

类比：
- Plugin = 给手机装新 SIM 卡（接入新基础设施）
- Skill = 教助手一个新口语任务（具体能做的事）

**接入新模型属于 code plugin**（需要运行时挂钩 provider），所以放在 `extensions/`，每个一个 npm 包。`skills/` 目录是另一个故事，后续单独讲。

## 用户复述与反馈

### Q1：OpenClaw 是个什么项目？

**用户复述**：
> "一个支持用户通过各种流行通信 app 通道，对接使用 AI AGENT，同时底层支持接入不同 AI 大模型的开源项目。"

**反馈**：✅ 框架级理解到位。抓住了三层结构（channels / agent / providers）。

**唯一遗漏的维度**：local-first / personal——OpenClaw 强调"runs on your own devices"，跑在用户自己设备上而非 SaaS。这是很多设计选择（沙箱、终端优先、安全偏执）的根因。

**升级版定义**：一个跑在用户**自己设备上**的个人 AI 助手，通过用户已经在用的聊天工具（24+ 个 channel）交互，底层可对接多种 AI 大模型，能做真实的事。

### Q2：核心设计哲学是什么？

**用户复述**：
> "通过 skill 来解决用户不同需求的 AGENT？"

**反馈**：⚠️ **plugin vs skill 的混淆复发**——这是本节课刚澄清的概念，但首次复述时仍混用。属于"认知粘性"，需在后续学习中刻意巩固 2-3 次方能彻底内化。

**正确答案**：**核心精简 + 插件化（Plugin-first / Core stays lean）**。主仓库只做骨架（路由、调度、安全边界），所有具体能力通过 plugin 插入。这是 `extensions/` 下有 126 个目录、`packages/` 下有 `plugin-sdk` 的根本原因。

**麦当劳类比**（用于巩固 plugin vs skill 区分）：
- Plugin / Extension = 餐厅里的**柜台机器**（点餐机、咖啡机）→ 基础设施
- Skill = 菜单上的**具体食物**（巨无霸、薯条）→ 具体能产出的东西
- Agent = **厨师**（编排）
- Channels = **门口传菜员**（与用户接触的窗口）

→ 加新 AI 模型 = 加一台点餐机 = Plugin（Extension）
→ 加"写诗"能力 = 加一道新菜 = Skill

### Q3：接入新 AI 模型放哪个目录？

**用户复述**：
> "EXTENSIONs 那个扩展单词的目录下加新目录来解决"

**反馈**：✅ 完整正确。用"扩展单词的目录"描述说明对 `extensions` 这个词的语义而非拼写已建立挂钩。
